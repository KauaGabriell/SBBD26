04 Relatório Técnico
Benchmark · Goiás 2016–2018
📋 Resumo
  • Região: Goiás, Brasil
  • Volume: ~680.506 polígonos
  • Por ano: 340.253 polígonos
  • Período: 2016 e 2018
  • Fonte: IBGE LULC
  • Operação: Interseção espacial + cálculo de área
📊 Resultados
  • PostGIS: 5,23 segundos
  • 5.236,86 ms (tempo médio)
  • GeoPandas: 62,28 segundos
  • 62.288,20 ms (tempo médio)
  • Speedup: 11,89× favorável ao SQL
  • Redução: 91,6% no tempo de execução
🔬 Análise dos Fatores
  • Princípio da localidade de dados: processamento onde os dados residem
  • Índice GIST (R-Tree): busca espacial O(log n) vs O(n)
  • Redução de I/O: sem transferência de dados para o processo cliente
  • Serialização evitada: Python carrega todos os dados em RAM
Conclusão do Estudo
"In-database (PostGIS) é a melhor abordagem para processamento geoespacial em larga escala."

Fatores Técnicos de Vantagem SQL
  • 🗂️
    Índice GIST (R-Tree)
    Filtragem espacial em O(log n) antes da operação de interseção
  • 💾
    Zero transferência de dados
    Python serializa e carrega ~680k polígonos na memória RAM
  • ⚙️
    Execução nativa C/C++
    PostGIS usa GEOS nativo; Python adiciona overhead de interpretação
  • 🔄
    Pipelining no SGBD
    PostgreSQL otimiza o plano de execução com estatísticas atualizadas
  • 📡
    Princípio da localidade
    Dados processados onde residem, sem custo de I/O de rede
